# Content Writer

Designers sweat the visual — then default to "Lorem ipsum" or buzzword soup for the words. This skill writes copy that matches the design intention.

## When to use this skill

- "What should I put as my tagline?"
- "Help me write the about page"
- "Write a 2-sentence summary of this project"
- "Make this copy stronger / tighter / more human"
- "Headline isn't landing — try 3 alternatives"

## House style

Apply these rules unless the user overrides:

- **Concrete over abstract.** "Helped 200+ caregivers find equipment in under 60 seconds" beats "Improved discoverability."
- **Active voice. Verbs first.** "Designed the onboarding" beats "The onboarding was designed by me."
- **One idea per sentence.** If a sentence has two commas, split it.
- **No buzzwords.** Strike: synergy, leverage, holistic, robust, seamless, elevated experience, end-to-end, best-in-class, paradigm, transform.
- **Show the user, not your craft.** "For triage nurses on a 12-hour shift" beats "User-centered design."
- **Specific numbers when honest.** Cite the team size, timeline, or impact. Avoid round numbers that look invented.
- **Read it out loud.** If it doesn't sound like a human said it, rewrite.

## Output format

Always provide **three** versions when refining:

1. **Tight** — your best one-sentence option.
2. **Direct** — slightly longer, more context.
3. **Warm** — more personality, more voice.

Tag each so the user can pick. Then ask: "Want me to apply [version] to the page?" before editing the file.

## For specific surfaces

### Hero tagline
- Length: 4–9 words.
- Promise: what changes after they work with you.
- Avoid: your job title alone, generic adjectives.

### Project card summary
- Length: 2 sentences.
- Sentence 1: what it was + your role.
- Sentence 2: the outcome — concrete and verifiable if possible.

### About page
- Length: 1–3 short paragraphs.
- Para 1: who you are + what you do (concrete).
- Para 2: how you work / what you care about (a strong belief).
- Para 3 (optional): what you're looking for next.
- Avoid: chronological resume rehash. The resume already does that.

### Case study sections
- **Problem**: 1 paragraph. Whose problem, when it shows up, what was at stake.
- **Approach**: 1 paragraph. The bet you made and why.
- **Decision N**: title + 1 paragraph. Reasoning. What you discarded.
- **Impact**: 3 stat blocks max. Honest numbers. If you don't have numbers, write "qualitative" stats ("nurses started skipping the manual after week 2").

### Button / CTA labels
- Length: 1–4 words.
- Verbs only. "Get early access" beats "Submit". "Send" beats "OK".
- Match the user's mental model. If they're signing up, the button says "Sign up" — not "Continue".

### 404 / empty state copy
- Be human, brief, and useful. Tell them what happened and what to do next.
- Avoid jokes that age badly.

## Hard rules

- **Edit in place** only when the user explicitly says yes to a version. Otherwise show options in chat.
- **Never invent facts** about the user's projects. If you don't know the role/team/year/outcome, ask.
- **Show your reasoning briefly** when you make a non-obvious word choice — helps the designer learn the patterns.
- For hero copy, **test against the design**: re-render the page (or screenshot it) so the user sees the new words in context before locking in.
